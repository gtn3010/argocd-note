ArgoCD Notifications to Microsoft Teams

Ishan Jain

8 hours ago4 min read

Overview

  • With ArgoCD notifications we can send notifications about application sync status in ArgoCD
  • It supports various notification services like email, MS Teams, custom webhooks etc
  • Since I tried to find good docs and blogs to send argocd-notifications to Microsoft Teams and didn't find any (Using the helm chart itself and not having to add annotation to each ArgoCD application), here is my version on how to do it

Terminology

Here I am not going to explain each and every component of argocd-notifications, I will let the official docs explain it better. Just to ensure smooth sailing while reading this , will put 101s on some main ideas about argocd-notfications

  • The notification trigger defines the condition when the notification should be sent. The condition is a predicate expression that returns true if the notification should be sent.
  • The notification template defines how the notification message would appear along with the message content in the configured notification service
  • The notification subscription is used to route the triggered notification along with its template to single/multiple notification service based on the triggers

Notifications workflow

  • ArgoCD notification continuously monitors all ArgoCd applications
  • When the triggers when condition is true, Using the notification template, the message is sent to the notification services configured in subscriptions

Prerequisites

  • Kubernetes cluster
  • GitOps workflows are set up for the cluster (ArgoCD + GitOps repository)
  • MS Teams channel

FAQ

Question: <channelName> referred in helm values Does this refer to the channel name or is this a keyword called channelName ?

  • Answer: <channelName> is just a placeholder, this can be changed to the actual channel name, For example - general

Question: How often do the alerts get evaluated? Where is that configured?

  • Answer: argocd-notifications continuously watches all applications.

Question: Does this work only for kind: application ?

  • Answer: This only works for kind: application . For Native kubernetes resources, sync failed/unknown/degraded will be caught as part of degraded health of the application that deployed the kubernetes resources

Question: Is it possible to route different alerts to different channels or have different sinks for different alerts?

  • Answer: Yes different alerts can be routed to different channels/sinks based on the triggers. For example, incase it is required to route Prometheus sync-alerts and MongoDB sync-alerts to different channels, we will have to create separate trigger(only the when condition will be different) for both.

Question: What would be the alerts lifecycle?

  • Answer: No lifecycle as these are just Teams/Email notifications, Incase the sink is pagerduty or opsgenie, the lifecycle can be maintained using those tools

Question: Where are these ArgoCD events generated from? The prometheus endpoints of the different components?

  • Answer: argocd-notifications has access to the applications CR in the namespace in which its is running (using its role ), so it directly monitors the applications.
