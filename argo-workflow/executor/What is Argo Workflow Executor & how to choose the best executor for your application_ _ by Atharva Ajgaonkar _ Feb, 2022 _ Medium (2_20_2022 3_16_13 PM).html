What is Argo Workflow Executor & how to choose the best executor for your application?

One of the key configurations in Argo Workflows is which executor to use. Choosing a Workflow executor is a really difficult choice that depends upon what your application demands whether its security, performance, stability, or some other special features, or all of them.

Don’t worry. It’s okay to get confused in this giant world of Argo. This blog post will solve your problem of selecting the best executor that your application demands. In this blog, we will take a smooth ride and visit every executor so tie up your seats. But first, let’s understand What is an executor.

What is an executor?

  • A workflow executor is a process that conforms to a specific interface that allows Argo to perform certain actions like monitoring pod logs, collecting artifacts, managing container lifecycles.

Now as we have got an idea about the definition of executor, let’s figure out which executors suit best for your application but before that let's get some more clarity on the security & performance of containers.

Security in containers:

  1. Docker containers are run with root privileges. Running your container as root can be useful in development mode but in production, any application breach can create disasters in your system like modifying system files, stopping or launching privilege access, changing user id or group id & so on.
  2. External application dependency also can inject vulnerability in containers.
  3. Non-root containers regulate which users in which namespaces are able to execute specific processes, what volumes can be accessed, and what ports your container is allowed to access.

Performance in containers:

  1. Performance of container is dependent upon CPU utilization, Memory consumption & Disk IO.
  2. The CPU benchmark can be measured concerning the time to perform a dedicated task.
  3. Memory benchmark measured by how much latency is present to perform Read/Write operation in memory

What type of executors does Argo Workflow offer?

EMISSARY MODE (default for > v3.2)

Argo teams suggest you use this mode. But this mode is not battle-tested as PNS mode, but on the flip to make stabilize faster Argo Team is actively collecting feedback and involved in bug fixing. This mode is for users less concerned with stability and backward compatibility

Security:

  1. This mode cannot escape the privileges of the pod’s service account.
  2. No privileged access
  3. Can runAsNonRoot.

Migrating to Emissary can become a small challenge because as we know Emissary containers always need “CMD” parameters in containers. So, if you are using containers like Kaniko then things can get complicated for you.

Docker executor is default in <= v3.2

We are all familiar with docker containers but we also know due to security concerts docker container runtime is deprecated in Kubernetes since v1.20

  1. It needs root access to run containers so the policy agent restricts it.
  2. It can escape the privileges of the pod’s service account
  3. Docker containers need to communicate with docker daemon if we need to build docker images inside a container, which exposes vulnerability. If one says let’s add a daemon on the Node level, which may cause things from bad to the worst due to which anyone can communicate with daemon and any process can request to start a container.

PNS Mode:

PNS is one of my favorite modes. Why? Because it's well tested and supports most of the applications.

But you must consider the following points before selecting this mode:

  1. Containers that run less than 15 sec doesn’t work
  2. It won’t even support windows images.
  3. No support for container set template
  4. Can runAsNonRoot, if you use volumes (e.g. emptyDir) for your output artifacts
  5. No privileged access
  6. cannot escape the privileges of the pod’s service account
  7. Information visible in /proc to processes running in other containers is protected only by regular Unix permissions,

Kubernetes API (k8sapi):

This mode is popular as well as well tested. Security is similar to PNS & Emissary:

  1. No privileged access
  2. Cannot escape the privileges of the pod’s service account
  3. Can runAsNonRoot
  4. If your application requires storage on the base image (in /tmp folder) then this mode requires extra volume as local storage. This mode is least scalable as Log retrieval and container operations are performed against the remote Kubernetes API

Conclusion: Argo workflow provides versatile executors. All in all, if your application requires high security and you are less concerned about backward compatibility then go with Emissary. If you want stability and confidence over your executors PNS would be the best choice. But if your application demands windows containers then go for Kubernetes API with some additional settings as per mode.
