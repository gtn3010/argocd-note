Automated branch deployment on Kubernetes

Boris Djurdjevic
Follow
Dec 30 · 4 min read

This post describes how to configure automated branch deployments on Kubernetes using Argo Events.

What are the use cases

In my previous work experience I saw the need of branch deployments for various reasons like

  • you want to have a branch continuously integration tested
  • the testing team wants to be able to test against a branch
  • the business wants to preview a feature on a deployed instance
  • you just want to have your code deployed to enjoy it

But at the end, each team or project solved the exact same problem using different ways which is generally not a problem but often results in extended complexity, orphaned deployments or security issues. Most of such strategies incorporate CI jobs that run a deployment using kustomize and revert it the same way. But on a shared cluster you struggle with the permissions of having enough rights to deploy and destruct all required resources but not having access to other deployments. But what if you could define your trusted deployment externally and use a service account to deploy it?

What is Argo Events

Argo Events is an event-driven workflow automation framework for Kubernetes. This allows us to ingest events when a branch is created or deleted on a VCS and to act on these events with creating and destroying deployments. This post doesn’t explain how to deploy Argo Events, as it’s easy to do so.

For the event source, where we will be listening to changes, we will choose Gitlab but Argo Events supports more than 20 EventSources.

For the action we take, we will use the Kubernetes Object Trigger as it can create and delete Kubernetes Resources. Our resource will be an ArgoCD Application which deploys a Helm Chart so we have our branch deployment also visible in ArgoCD and can use a trusted Helm Chart for the deployment. For the ease of usage and example, we decide that we will only deploy branches that have an MR open.

Configuring EventSources

Having Argo Events setup, we can define a gitlab EventSource that registers itself for MergeRequest events on the defined gitlab project:

You can see that we connect the “microfast/website” project with this source and configure a webhook for it. You must create an according ingress for the webhook. Additionally we have a gitlab-access Secret that holds the gitlab personal access key of the user that has the rights to see merge requests and a gitlab-secret which is a random string to verify the webhook.

After creating this resource, Argo Events will configure the defined projects automatically to send webhooks to the defined URL so we can react when something is happening with merge requests.

Configuring Triggers

Now as we got the “gitlab” EventSource running, we are receiving events from Gitlab to our Argo Events instance and we just need to react on them. Therefore we need Sensors that handle the deployment and undeployment. Currently it’s not possible to have a dependency/condition construct to have a single Sensor for deployment and undeployment so we will have to define two of them.

The deployment Sensor listens on our gitlab events and filters only the ones that are resulting in an active MR state (open, reopen, update). Then it defines the trigger to create a Kubernetes resource of the type of an ArgoCD Application and fills the dynamic parameters with the Merge Request ID.

You can see that the trigger object is creating a helm chart deployment which sets the variable “MR” to the Merge Request ID, which is used by the helm chart to create unique global resources such as Ingress URLs. Also the ArgoCD Application name and the custom namespace has the ID included to prevent conflicts. After the instantiation of this resource, ArgoCD runs the automated sync so we can soon access the deployed application.

For the undeployment case, we create another Sensor that reacts only to closing and merging events and just deletes the ArgoCD Application, which also removes the helm based deployment:

But we also need to customize our deployment and image builds to deploy the release we want to see.

For the CI job this means in our case that we build a separate image, which we label accordingly with mr-${CI_MERGE_REQUEST_IID} only on merge requests.

As for the deployment we chose to use helm charts as we can easily propagate a discriminator variable to render unique global resources or identifiers. This can be for example an Ingress URL or the image label:

Further implementation

You can use this example as a starting point. In order to scale it’s recommended to create RBAC rules and a dedicated service account for the Sensor to create and delete the Kubernetes resources.
